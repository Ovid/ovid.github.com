The AI Tramp Freighter

Tramp Freighter Blues is a game that I wrote entirely with AI. I reviewed the specs and the plans. I did not read the code. I did not review the code. After months of work, I felt I was finally at the point where I could build larger AI systems using only English as my programming language. It’s roughly 30,000 lines of React and TypeScript. I don’t know React or TypeScript.

You Don’t Debug with an Oscilloscope

There are those who who will sneer at my 4+ decades of programming in (rough order of learning them) BASIC, Assembly, C, UniBASIC, COBOL, VBA, EasyTrieve, JavaScript, Perl, Java, Prolog, Python, and probably some that I’ve forgotten about, and tell me that I obviously don’t understand programming. In reality, using AI to program is a new discipline, requiring new skills, but those who refuse to learn these new skills create AI slop and then blame the AI. It will likely be several years before we’re past this.

So I’ll ignore them. I’m talking to you.

In in 1940s, programming mean rewiring a computer. Debugging often meant using an oscilloscope or using a voltmeter to find a blown vacuum tube. Computer science meant having an electrical engineering degree.

Punch cards were revolutionary, letting people store assembly, and later, high-level langauges like FORTRAN and COBOL, separately from the computer.

By the 1970s, C arrived, marking one of the first truly portable system languages.

Today, languages like Perl and Java would have been science fiction in the 1940s. Today, they’re just another tool in the toolbox. If their output is buggy, very few who use them would ever bother to check the machine code; they’d just rewrite the real source of truth: the programming language.

The next leap is already here: English (or whatever language you speak) is becoming the source code. The product is still the source of truth

Note: two years ago I wrote about building an iPhone app with ChatGPT. That post is already laughably out of date.

With AI, we moved from autocompleting methods to building entire apps. I don’t mean “vibe coding.” I gave that up quickly because it’s miserable. I’ve also used Spec-Driven Development (SDD), There are various ways of interpreting SDD, so I’ll skip that argument. Instead, I’ll say that I’ve spent much of the past year on a quest: how well can I build larger software using these AI tools? My first few attempts were failures, but I kept getting further each time.

I don’t really have a choice in this. My role involves uplifting the technical skills of 200+ developers and that includes AI. If they’re turning out slop, I’ve failed.

Over this year, I’ve finally narrowed down the AI coding issues to four core areas. I built a 30K-line game to find out if I’m right. The current work is a greenfield project and 30K is a mid-sized project, not a big one, so people will still be dismissive, but I’m seeing developers work on brownfield projects with great success. More on that later.

Note that for all of the following prompts, run them in a fresh session. The less context you use, the better your results will be.

Also, regardless of whether you’re using a CLAUDE.md file, Kiro Steering or Spec Kit Constitution files, review them and make sure they’re both correct and small. If they don’t match your requirements, my prompts will not save you.

The AI Coding PAAD

The PAAD is my model of the issues with AI coding. “All models are wrong; some are useful.” Mine, I’ve found, is useful. By clarifying what the problems are, I’ve been able to systematically tackle these problems one by one. This has come from tons of personal experience, reading where others are struggling, and discovering that all AI systems fall prey to these problems.

Pushback Against the Yes Men

By now, you’ve certainly heard about the sycophancy problem with AI. You ask it to build a web site to sell a blockchain-based loyalty program for a lemonade stand and it tells you you’re brilliant!

The worse problem, though, is that it tends to be a “yes man.” Sycophants are yes men, but they’re also constantly telling you how amazing you are. “Yes men,” by contrast, don’t try to butter you up, but they’ll do what you say without question. This is not the AI you want.

Recently, a development team was struggling to get good results from AI programming, so I pair programmed with one of them. As soon as he created his spec, he was about to move onto the plan. I stopped him and asked if he had heard about my “clarification prompt.” (I call it “clarification” in the office because it sounds nicer than “pushback”)

He hadn’t, so I sent it to him and asked him to run it.

The pushback prompt looks at your spec and searches for obviously missing requirements, contradictory requirements, bad ideas, and so on. Only once have I had this prompt not catch a problem with my specs. This developer has written a very careful spec, so I was surprised that the prompt only found a single issue, but it was a doozy. Seems someone had just pushed a feature that meant his spec was wrong. Without the prompt, Kiro would have happily built something that didnt’t work and he would have understandably blamed Kiro!

The requirements.md file has just been edited.
Please analyze it for:

1. Obvious omissions - missing requirements
   that should be included based on the context
   of this project
2. Ambiguous requirements - requirements that need
   clarification or are unclear
3. Contradictory requirements
4. Security concerns - any requirement that might
   present a security issue
5. A quick check in the code for anything which
   might conflict with the requirements
6. Any requirements not in EARS format

If you find issues:

- Ask the user ONE question at a time
- Present specific options, from best to worst,
  with your recommendation for fixing the issue,
  along with a short explanation
- Wait for their answer before asking the next
  question
- After all questions are answered, update the
  requirements.md file with the clarifications
  and additions based on the user’s responses

If no issues are found, inform the user that
the requirements look complete and clear.

Note: running the pushback prompt several times until you get “complete and clear” at least twice is a good idea.

Alignment

But you’re a diligent developer, right? The one I was pair programming with is. He knew that for SDD, you have to write an amazing spec. We’re not vibing here; we want to ensure the AI writes exactly what you want. So what if you have written a perfect spec? (I’ve only once had the pushback prompt not catch something). Well, you still have problems and they’re not your fault.

In some versions of SDD, you write a spec and the AI builds a plan and then creates the tasks for it. So our developer followed that process and I asked him to run my “alignment” prompt.

Click to see an alignment prompt 

Analyze the requirements, design, and implementation tasks
for alignment.

Perform the following checks:

1. **Requirements Coverage**: Verify that every requirement
has corresponding tasks in the implementation plan
2. **Scope Compliance**: Ensure no tasks exist that fall
outside the defined requirements
3. **Design Alignment**: Check that the design properly
addresses all requirements and that tasks implement the
design

For each issue found, provide:

- The specific requirement/design/task that is misaligned
- The nature of the misalignment (missing coverage, out of
scope, design gap)
- Concrete remediation steps (update requirements, add
tasks, remove tasks, clarify design)

Present findings ONE-BY-ONE, with recommendations in order
from best recommendation to least, in a clear, actionable
format with severity levels (Critical, Important, Minor).
Give options and a recommendation. WAIT for me to give
guidance before moving to the next finding.

Review these files:

- Requirements documents: **/requirements.md
- Design documents: **/design.md
- Implementation/task documents: **/tasks.md

When I have confirmed that the final alignment is complete,
rewrite all of the tasks in red/green/refactor format if they
are not already in that format.

The alignment prompt walks through the requirements, one by one, and ensures that every requirement has a task and no tasks exist which are not in the requirements. You would think that with using the most powerful foundation models that they would create tasks matching exactly what you asked for, no more, no less. I have never had an alignment check succeed on the first try. The developer I was pairing with was working through problem after problem, correcting all of the alignment issues the prompt found. W

Without alignment, the AI will build something that is not what you asked for!

As with the pushback hook, running alignment more than once is recommended.

As an aside, the when alignment is complete, my prompt instructs the AI to rewrite tasks in red/green/refactor format. Why? This tends to create more tasks, but smaller tasks. Do each task in a fresh context window. Red/green/refactor is a common convention that the AI will understand.

Further, the “red” phase tells it to write the test first. If that test passes, or fails in an unexpected way, the AI will generally look into it and this often saves you some grief.

The “green” phase means “write the simplest code to make the test pass.” That tends to keep your code, well, simpler.

The “refactor” phase often leads to the AI cleaning up unused variables or imports, or sometimes discovering that it’s duplicated existing functionality. Small wins like this will help keep your codebase in better shape.

Architecture

For a greenfield project, AI often starts out with a decent architecture, or if it’s off, it’s easier for you to see that. But as time goes on, architecture starts to ... drift.

That tiny class becomes a god object. Logging is ad hoc or non-existent. try blocks with an empty catch. Dead code, business logic in the UI, poor transactional boundaries and more!

This kills many developers. AI starts out great, but over time, it creates a sloppy mess that is impossible to maintain. This is the wall I was hitting in my previous attempts to go beyond “subroutine autocomplete” to “writing entire apps” (all while not reading the code).

As part of my experiments in how to solve this, I built Ọya , a CGRAG system that’s conceptually similar to DeepWiki , but without all of the pages for your repo. Instead, you just ask it questions. Lately, however, I’ve been building Shesha , an RLM recursive language model . That effectively gives any LLM unlimited context with better long-context reasoning and it’s been doing an amazing job of highlighting architectural issues. You can chat with it just like a normal LLM and throw huge codebases at it.

Shesha is great and RLMs can give you excellent long-context reasoning over massive sets of documents (I’ll be releasing a “Document Explorer” soon. Just upload tons of docs and query them). But I wanted something that wouldn’t take me too much out of my workflow. So with the release of Claude Opus 4.6, I created a Claude Skill that is special-purposed for this task. (It’s a massive prompt).

Note: this was thrown together _very_ quickly and desperately needs optimization. However, it’s also worked fairly well, so I haven’t returned to it for optimizating. Use at your own risk!


---
name: ovid-architecture
description: >
  Analyze a codebase for software architecture strengths
  and flaws and report the good and the bad.
--


Architecture Strengths & Flaws Analyzer


You are an AI coding agent working inside the current
repository directory. Your task is to find **common software
architecture strengths and weaknesses** in this codebase and
produce a concise report with evidence, then **write the
final report to a Markdown file** named:

\--architecture-report.md

* Use today’s date in ISO format: YYYY-MM-DD
* Determine \ from the git remote or
top-level folder. If this directory is not a git repo, omit
the repo name portion and write:
\-architecture-report.md.

> If you must keep the original filename for compatibility,
use:
> `--architecture-flaws.md` (and
keep the title “Architecture Flaws Report”).
> Otherwise prefer the neutral “architecture-report” naming
above.


Goal


Produce a balanced report that highlights:

* **Strengths** (what’s working architecturally and why it matters)
* **Flaws/Risks** (architectural problems and their likely impact)

**Do NOT propose fixes yet.** This is diagnosis only.


Scope


Include findings across **all** of the following
architectural **issue types** (prioritize high-signal, but
ensure coverage across the set). For each type:

* If **observed**, include at least one evidence-backed
finding (or reference a finding ID).
* If **not found**, mark **Not observed**.
* If tooling limits prevent assessment, mark **Not
assessed**.

Flaw/Risk types

1. Global mutable state — Shared state that any code can
change, making behavior unpredictable and hard to test.
2. God object — One class/service accumulates too many
responsibilities and becomes a brittle dependency magnet.
3. Tight coupling — Components depend on concrete details of
each other, so small changes ripple widely.
4. High/unstable dependencies — Core modules depend on
“leaf” modules, forcing rebuilds and coordinated releases.
5. Circular dependencies — Packages/modules import each
other, complicating builds, testing, and refactoring.
6. Leaky abstractions — An abstraction requires callers to
know underlying details to use it correctly.
7. Over-abstraction — Too many layers/interfaces for
uncertain future needs, increasing complexity without
payoff.
8. Premature optimization — Architecture choices made for
performance before evidence, harming clarity and
flexibility.
9. Shotgun surgery — A single logical change requires edits
across many files/services.
10. Feature envy / anemic domain model — Business logic
lives in services/utilities while domain objects are just
data bags.
11. Low cohesion — Modules group unrelated behaviors, making
boundaries unclear and changes risky.
12. Hidden side effects — Functions/methods do more than
their signature suggests, surprising callers.
13. Inconsistent boundaries — Responsibilities drift between
layers/services, causing duplication and confusion.
14. Distributed monolith — “Microservices” in name only,
with heavy synchronous coupling and shared release
constraints.
15. Chatty service calls — Too many small network calls
between services, increasing latency and failure surfaces.
16. Synchronous-only integration — Everything depends on
immediate responses, turning partial outages into full
outages.
17. No clear ownership of data — Multiple services write the
same data, creating conflicts and integrity problems.
18. Shared database across services — Services couple
through schema and queries, making independent evolution
difficult.
19. Lack of idempotency — Retries create duplicates or
corruption because operations aren’t safe to repeat.
20. Weak error handling strategy — Errors are swallowed,
over-generalized, or inconsistently surfaced.
21. No observability plan — Missing logs/metrics/traces
makes debugging and capacity planning guesswork.
22. Configuration sprawl — Behavior is controlled by
scattered configs/flags with unclear precedence and drift.
23. Dependency injection misuse — DI becomes a maze of
indirection that obscures control flow.
24. Inconsistent API contracts — Endpoints/events evolve
without compatibility discipline, breaking consumers.
25. Business logic in the UI — Critical rules live in
front-end code, leading to duplication and inconsistent
behavior.
26. Poor transactional boundaries — Operations span multiple
systems without a strategy, leaving partially-updated
states.
27. Temporal coupling — Components must be called in a
specific order/timing to work correctly.
28. Magic numbers/strings everywhere — Important values are
hard-coded and repeated, making change error-prone.
29. “Utility” dumping ground — Generic helper modules grow
into unowned, untestable grab-bags of unrelated code.
30. Security as an afterthought — AuthZ/authN, secrets, and
trust boundaries bolted on late/inconsistently enforced.

Additionally, you MUST include these concrete issues:
31) Dead code / unused dependencies — Increases cognitive
load and attack surface.
32) Missing or inadequate test coverage for critical paths —
Architectural risk that compounds all the others.
33) Hard-coded credentials or secrets in source — Concrete
security flaw; call out separately when found.
34) Inconsistent error/logging conventions across services —
Specifically cross-service inconsistency (formats, levels,
fields, correlation IDs).

---


Strength categories


In addition to flaws, explicitly look for **architecture
strengths**. Report strengths using the same evidence
standard as flaws (label, explanation, evidence). Assess at
least these categories:

S1) Clear modular boundaries — Well-defined
packages/modules/layers with minimal leakage.
S2) High cohesion — Modules/services group related
responsibilities; boundaries feel “natural”.
S3) Loose coupling — Abstractions/interfaces/events reduce
ripple effects from change.
S4) Dependency direction is stable — Core depends on stable
contracts; leaf depends on core; minimal “core depends on
leaf”.
S5) Dependency management hygiene — Minimal circular deps;
consistent import conventions; sensible package structure.
S6) Consistent API contracts — Versioning/compat discipline;
schema validation; backward-compatible changes.
S7) Robust error handling — Consistent error taxonomy;
errors surfaced appropriately; avoids swallowing exceptions.
S8) Observability present — Structured logs, metrics,
traces, correlation IDs, health checks.
S9) Configuration discipline — Centralized config; clear
precedence; safe defaults; separation by environment.
S10) Security built-in — AuthN/Z patterns, secret
management, least privilege, explicit trust boundaries.
S11) Testability & coverage — Tests around critical paths;
good seams; determinism; contract tests where appropriate.
S12) Resilience patterns — Timeouts, retries with
idempotency, circuit breakers/backpressure, async
integration where appropriate.
S13) Domain modeling strength — Business logic lives with
domain entities/value objects; invariants enforced close to
data.
S14) Simple, pragmatic abstractions — Abstraction level
matches current complexity; avoids over/under engineering.

> If a strength category is “not applicable” due to repo
nature (e.g., no networked services), mark it **Not
applicable** and briefly explain.

---


Constraints


* **Do NOT propose fixes** (no refactors, no “should do X”).
Only describe strengths and weaknesses with evidence.
* Prefer high-signal findings over exhaustive listing, but
ensure coverage across the required sets.
* Every finding MUST include:
  (1) short label,
  (2) 1–2 sentence explanation,
  (3) concrete evidence (file paths + symbol names + a small
  excerpt or line range reference).
* Validate candidates by opening files to reduce false
positives.

---


Process


A) Repo identification

* Detect if this is a git repo (`git rev-parse`).
* If yes, determine repo name:

  * Prefer: basename of top-level directory OR derive from
  `git remote get-url origin` (strip .git, take last path
  segment).
* Set output filename accordingly.

B) Repo overview

* Identify primary languages/frameworks and key directories
(apps/, services/, packages/, src/, lib/, etc.).
* Estimate size: number of services/modules/packages if
applicable.

C) Dependency & structure analysis

* Build a quick dependency picture:

  * top-level modules/packages
  * identify potential cycles via import graphs or simple
  heuristics
  * identify “core depends on leaf” patterns
* Also note **positive structure signals** (clean layering,
bounded contexts, stable interfaces).

D) Search strategy

* Use ripgrep/git grep and lightweight heuristics;
optionally use AST tools if present.
* Specific searches (risks + strengths):

  * Global state: module-level mutable variables,
  singletons, service locators, static mutables.
  * God objects: very large classes/files,
  managers/services/controllers with huge responsibility
  surface; high fan-in/fan-out.
  * Coupling/cycles: cross-layer imports, concrete
  instantiations, circular imports.
  * Boundaries/cohesion: directory organization, naming,
  cross-layer leakage, “one reason to change”.
  * Abstractions: leaky vs clean; over-abstraction vs
  pragmatic, consistent interfaces.
  * Side effects: IO/DB/network/event publishing in
  “pure-looking” functions.
  * Error handling/idempotency/resilience: catch-and-ignore,
  blanket exceptions, retries with idempotency keys,
  timeouts, circuit breakers/backpressure (if present).
  * Observability: structured logging, tracing, correlation
  IDs; consistency across services.
  * Security: authN/Z patterns, secret management;
  hard-coded secrets.
  * Dead code/unused deps: unused packages, unused
  files/modules, unreachable code paths, deprecated
  directories, stale feature flags.
  * Tests/coverage: identify critical paths (entrypoints,
  handlers, core domain services) and check for
  corresponding tests; highlight strengths and gaps.

---


Output format (Markdown)


1. Title: “Architecture Report — \”
2. Date: \
3. Repo overview (languages, key directories)
4. Strengths (ranked High/Medium/Low impact), 5–15 items,
each formatted exactly:
   * [Impact: High|Medium|Low]  — <1–2
   sentence explanation>.
     Evidence: \:\ (\), excerpt: "\"
5. Flaws/Risks (ranked High/Medium/Low impact), 10–25 items,
each formatted exactly:
   * [Impact: High|Medium|Low] \ — \<1–2
   sentence explanation\>.
     Evidence: \:\ (\), excerpt: "\"
6. Coverage checklist
   * **Flaw/Risk types 1–34**: Observed / Not observed / Not
   assessed, with 1 short line and optional pointer to a
   finding ID.
   * **Strength categories S1–S14**: Observed / Not observed
   / Not assessed / Not applicable, with 1 short line and
   optional pointer to a finding ID.
7. Hotspots: top 3 files/directories to review (brief why;
can include both risk hotspots and “strong core” hotspots)
8. Next questions (max 5) to guide humans (questions only;
no suggested solutions)

---


Execution


* Perform the investigation now using available tools (`rg`,
`git grep`, `ls`, `tree`, language-specific linters if
already configured).
* Then write the Markdown file to the repository root with
the required filename.
* Finally, print the path to the generated file and a brief
summary (3–6 bullet points) of the highest-impact strengths
and risks.

Degradation

  • A broad category: security holes, weird edge cases at scale, subtle rot
  • Software quickly hits complexity levels where edge cases pop up constantly and the AI can’t see them
  • Humans struggle to articulate the broad patterns here, so it’s hard to communicate them to AI
  • Anthropic’s zero-day bug work is promising, but cost remains a question

3. The Historical Analogy (Expanded)

  • 1950s–70s: “Full stack” meant physical reality — no OS, no abstraction, assembly mapped 1:1 to CPU operations
  • The shift: high-level languages emerged → we stopped caring about hardware → EE split from CS
  • Today: we write Python/Perl/Java and trust the compiler. Nobody reviews the machine code output
  • Tomorrow: our children will find it quaint that we typed while loops into a terminal
  • The progression: Autocomplete → Vibe Coding → Full Apps
  • Programming isn’t going away — but the “electrical engineering” layer of it is. The source of truth shifts upward again

4. The Experiment: Building a Game Without Reviewing Code

Why a Game?

  • Most software has isolated components (registration, cart, search) communicating across well-defined boundaries
  • Games are different: actions have ripple effects across everything. This stress-tests AI coding in ways a typical CRUD app doesn’t
  • Chose the most complicated scenario from AI-generated options — not because it was hardest, but because it was most interesting

The Approach

  • Original intent: how far can I push AI without writing or reviewing code?
  • Started by reviewing AI code obsessively, but saw an easier path as quality improved
  • Used Claude for scenario generation (best at writing rich, interconnected scenarios). Gemini and ChatGPT produced “flat” results
  • Long session to refine the vision, then broke it into 8 stages (some later broken into substages)
  • Practical rule: keep context under 50%. Quality drops after that
  • Desktop browser only — not mobile

8. The Uncomfortable Question

  • “I can often see errors the AI is fixing and I can guide it because I’ve been programming for decades.”
  • How long before that skill isn’t needed?
  • The interstitial period — where human expertise is still essential but visibly shrinking — is going to be painful
  • This isn’t abstract. It’s happening now, to people reading this article

9. Closing: The Source of Truth Shifts Again

  • Circle back to the historical analogy
  • EE → high-level languages → AI: each shift moved the “source of truth” up one layer of abstraction
  • Programming isn’t dying. The layer is changing
  • PAAD is what stands between “AI can write code” and “AI can build software”
  • Current status: personal projects are largely working. The codebases grow, they tend to work, but with quirks. That gap is closing
